# Генератор отмазок

Одностраничный веб-сервис, где пользователь выбирает категорию ситуации, вводит причину и опциональный контекст о себе, а AI генерирует правдоподобную отмазку. После генерации отдельный LLM-грейдер оценивает правдоподобность по шкале 0–100, добавляет комментарий и риск.

**Стек:** FastAPI + OpenAI Python SDK (backend), NiceGUI (frontend).

---

## 1. Контекст задачи

Исходное задание — сделать одностраничный сервис «Генератор отмазок»:

- пользователь выбирает ситуацию (категорию);
- описывает, почему так получилось;
- при необходимости добавляет краткий контекст о себе;
- сервис генерирует текст отмазки;
- показывает рейтинг правдоподобности, комментарий и риск;
- даёт возможность скопировать готовый текст.

Полный PRD подготовлен отдельно, поэтому в README приводится только краткое описание назначения и технической реализации проекта.

---

## 2. Почему MVP без БД

База данных, авторизация и история генераций **намеренно не добавлялись**:

- цель — быстро собрать рабочий прототип под конкретную задачу;
- backend **stateless**: каждый запрос обрабатывается независимо;
- контекст пользователя (`person_context`) используется только в рамках текущего запроса и нигде не сохраняется;
- история прошлых отмазок не ведётся;
- категории ситуаций заданы в коде backend (список в `app.py`), без отдельного хранилища.

Это упрощает развёртывание и отладку MVP, но ограничивает персонализацию и аналитику на уровне продукта.

## 3. Пользовательский сценарий

1. Пользователь открывает страницу frontend (`http://127.0.0.1:8080`).
2. Frontend загружает категории через `GET /api/categories`.
3. Пользователь выбирает категорию в выпадающем списке.
4. Пользователь вводит причину («Почему так получилось»).
5. Пользователь опционально вводит контекст о себе.
6. Нажимает **«Сгенерировать отмазку»**.
7. Frontend отправляет `POST /api/generate` на backend.
8. Backend через `ExcuseService` вызывает **ExcuseGenerator** (текст отмазки).
9. Backend вызывает **PlausibilityGrader** (оценка, комментарий, риск).
10. Frontend отображает результат: отмазку, шкалу правдоподобности, комментарий и риск.
11. Пользователь копирует текст отмазки кнопкой «Скопировать».

## 4. Архитектура проекта

```
Frontend (NiceGUI)
        ↓
FastAPI backend
        ↓
ExcuseService
        ↓
ExcuseGenerator  →  ExcuseDraft
        ↓
PlausibilityGrader  →  ExcuseEvaluation
        ↓
ExcuseResult
        ↓
Frontend (карточка «Результат»)
```

Два LLM-вызова разделены: генератор не оценивает ответ, грейдер не переписывает отмазку.

## 7. Сущности и классы

### Pydantic-модели (`schemas.py`)

| Модель | Роль |
|--------|------|
| **ExcuseRequest** | Вход POST `/api/generate`: категория, причина, опциональный контекст |
| **ExcuseDraft** | Промежуточный результат генератора: только `excuse` |
| **ExcuseEvaluation** | Промежуточный результат грейдера: `plausibility`, `comment`, `risk` |
| **ExcuseResult** | Финальный ответ API для frontend |
| **Category** | Одна категория (`id`, `name`) |
| **CategoriesResponse** | Обёртка списка категорий для GET `/api/categories` |

### Сервисы (`services/`)

| Класс | Роль |
|-------|------|
| **ExcuseGenerator** | Первый LLM-вызов: генерация текста отмазки |
| **PlausibilityGrader** | Второй LLM-вызов: оценка правдоподобности |
| **ExcuseService** | Связывает generator и grader, возвращает `ExcuseResult` |

### Вспомогательные модули

- **config** — `setup_env()`, `log_openai_key_status()`, `configure_logging()`
- **openai_client** — `get_openai_client()` с проверкой ключа

## 8. Проектирование схем

Валидация на уровне Pydantic (и FastAPI для HTTP):

| Поле | Правила |
|------|---------|
| `category` | обязательна, 1–80 символов |
| `reason` | обязательна, **минимум 3 символа**, максимум 300 |
| `person_context` | опционально, максимум 700 символов |
| `excuse` (в draft/result) | минимум 5 символов |
| `plausibility` | целое число **от 0 до 100** |
| `comment` | минимум 3 символа, максимум **180** (`ExcuseEvaluation` и `ExcuseResult`) |
| `risk` | опционально, строка до **180** символов; нормализуется в `PlausibilityGrader`, те же лимиты в `ExcuseResult` |

Некорректный `reason` (например, `"ab"`) даёт ответ **422 Unprocessable Entity** до вызова LLM.

## 9. API

Базовый URL backend: `http://127.0.0.1:8000`

### `GET /api/categories`

Возвращает список доступных категорий.

**Пример ответа:**

```json
{
  "categories": [
    {"id": 1, "name": "Опоздание"},
    {"id": 2, "name": "Не сделал задачу"},
    {"id": 3, "name": "Забыл событие"},
    {"id": 4, "name": "Не ответил вовремя"},
    {"id": 5, "name": "Не пришёл на встречу"}
  ]
}
```

### `POST /api/generate`

Генерирует отмазку и оценивает её.

**Тело запроса:**

```json
{
  "category": "Опоздание",
  "reason": "проспал, потому что ночью доделывал проект",
  "person_context": "я студент, учусь и работаю"
}
```

`person_context` можно передать как `null` или опустить.

**Пример ответа:**

```json
{
  "excuse": "Извини за опоздание, ночью доделывал проект и проспал. Уже выхожу.",
  "plausibility": 74,
  "comment": "Отмазка звучит естественно для студента и логично объясняет опоздание.",
  "risk": "Могут усомниться, если такие опоздания повторяются часто."
}
```

**Пример ошибки валидации (422):**

```json
{
  "detail": [
    {
      "type": "string_too_short",
      "loc": ["body", "reason"],
      "msg": "String should have at least 3 characters"
    }
  ]
}
```

Интерактивная документация: http://127.0.0.1:8000/docs

## 10. LLM-подход: Generator + Grader

Генерация и оценка **разделены на два независимых шага**:

1. **Generator** — только текст отмазки, без JSON и без оценки.
2. **Grader** — только анализ уже готового текста.

**Почему так:**

- проще контролировать промпты и параметры (температура, лимиты токенов);
- грейдер может использовать **structured output** (`responses.parse()` + схема `ExcuseEvaluation`);
- при сбое грейдера endpoint не падает: возвращается **fallback-оценка** с нормализацией полей.

| Шаг | Модель | API | Параметры |
|-----|--------|-----|-----------|
| Generator | `gpt-4o` | `responses.create()` | `temperature=0.75`, `max_output_tokens=220` |
| Grader | `gpt-4.1` | `responses.parse()` | `temperature=0.1`, `max_output_tokens=300` |

## 11. ExcuseGenerator

**Вход:** `ExcuseRequest` — `category`, `reason`, `person_context` (опционально).

**Выход:** `ExcuseDraft` с полем `excuse`.

**Поведение:**

- генерирует короткую отмазку на русском (2–4 предложения);
- возвращает **plain text**, не JSON;
- **не оценивает** правдоподобность;
- в промпте запрещено выдумывать факты, которых нет в причине и контексте;
- если `person_context` не указан, не придумывает возраст, работу, учёбу и т.п.;
- при пустом ответе модели подставляется короткий безопасный fallback-текст.

---

## 12. PlausibilityGrader

**Вход:** исходный `ExcuseRequest` + `ExcuseDraft` (готовая `excuse`).

**Выход:** `ExcuseEvaluation` — `plausibility`, `comment`, `risk`.

**Поведение:**

- оценивает по шкале **0–100** с учётом категории, причины, контекста и текста отмазки;
- в промпте заложена шкала (слабая / нормальная / сильная отмазка), без автоматической «85»;
- использует `responses.parse(..., text_format=ExcuseEvaluation)`;
- **`_normalize_evaluation()`** — подрезка длинных `comment`/`risk`, clamp `plausibility`, замена пустых полей;
- **`_fallback_evaluation()`** — при ошибке API, пустом `output_parsed` или `ValidationError`; endpoint остаётся рабочим.

## 13. Frontend на NiceGUI

Отдельный процесс: `src/app/frontend.py` (порт **8080**).

**Особенности:**

- одна страница, без роутинга;
- карточка **«Параметры»** (категория, причина, контекст, кнопка генерации);
- карточка **«Результат»** (отмазка, шкала, комментарий, риск);
- категории с backend (`GET /api/categories`); в коде есть **FALLBACK_CATEGORIES** только если backend недоступен;
- `POST /api/generate` через **httpx** (запросы идут с сервера NiceGUI, не из браузера);
- кнопка копирования → `ui.clipboard.write()` + уведомление;
- **desktop:** две колонки (`grid-cols-1 md:grid-cols-2`);
- **mobile:** одна колонка, компактные отступы;
- стиль — спокойный **Retool-like** internal tool (серо-белые карточки, toolbar, без «лендинговых» эффектов).

## 14. Запуск проекта

### Требования

- Python ≥ 3.11
- [uv](https://github.com/astral-sh/uv)

### Установка зависимостей

```bash
uv sync
```

### Переменные окружения

```bash
cp .env.example .env
```

В `.env`:

```env
OPENAI_API_KEY=your_openai_api_key_here
```

Настоящий ключ хранится только в `.env`. Файл `.env` **не коммитится**.

### Backend

```bash
uv run uvicorn app.app:app --reload --app-dir src
```

- API: http://127.0.0.1:8000  
- Swagger: http://127.0.0.1:8000/docs  

### Frontend

В **отдельном** терминале (backend должен быть запущен):

```bash
uv run python src/app/frontend.py
```

- UI: http://127.0.0.1:8080

## 17. Важные ограничения

- это **MVP без БД**;
- **нет авторизации** и учётных записей;
- **нет хранения** персональных данных и истории запросов на сервере;
- **API key** только в `.env`, не в коде и не в Git;
- файл **`.env` не коммитить** — в репозитории только `.env.example`;
- оценка правдоподобности зависит от модели и промптов; fallback не заменяет полноценную оценку при сбоях LLM.
